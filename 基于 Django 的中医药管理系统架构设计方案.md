# 基于 Django 的中医药管理系统架构设计方案

## 1. 设计目标

结合开题报告，本系统面向中小型中医药机构的日常管理场景，目标是解决传统管理方式中信息分散、纸质记录效率低、库存难追踪、权限边界不清等问题，实现中医药业务的数字化、规范化与可追溯化管理。系统设计应满足以下目标：

1. 支持中药材信息的标准化录入、维护与查询；
2. 支持方剂配伍、组成药材及用量管理；
3. 支持库存入库、出库、盘点、预警与流水追踪；
4. 支持管理员、药剂师、医护人员等多角色权限控制；
5. 保证系统具备良好的可扩展性、安全性与维护性。
   这些目标与开题报告中的研究内容和功能边界保持一致。

------

## 2. 总体架构设计思路

本系统采用 **B/S（Browser/Server）模式** 和 **前后端分离架构**。前端负责页面展示与交互，后端负责业务逻辑处理与数据访问，数据库负责业务数据持久化存储。整体技术路线如下：

- 前端：Vue.js
- 后端：Django + Django REST Framework
- 数据库：MySQL
- 认证机制：JWT
- 部署方式：Docker Compose 容器化部署
  这一路线与开题报告中确定的技术栈和系统设计阶段目标一致。

## 3. 分层架构设计

为了保证系统的清晰性与可维护性，本系统采用五层逻辑架构。

### 3.1 用户层

用户层是系统的直接使用者，主要包括：

- **管理员**：负责用户管理、角色分配、基础数据维护、系统配置；
- **药剂师**：负责药材信息维护、方剂维护、库存出入库操作；
- **医护人员**：负责方剂查询、药材查询、库存查看等。
  该角色划分来源于开题报告中的多角色设计思路。

### 3.2 表示层

表示层由 Vue.js 构建单页面应用，负责：

- 页面路由管理；
- 表单输入与数据校验；
- 列表分页、筛选、搜索；
- 图表展示与预警提示；
- 用户操作反馈。
  根据开题报告，前端页面至少应覆盖登录页、药材管理列表、方剂详情页、库存预警页等。

### 3.3 接口服务层

接口服务层由 Django REST Framework 提供 RESTful API，负责：

- 接收前端请求；
- 返回 JSON 格式数据；
- 统一参数校验与异常处理；
- 提供认证、查询、增删改查接口。
  这部分与报告中“制定 API 交互规范，明确请求方式、参数及返回格式”的要求一致。

### 3.4 业务逻辑层

业务逻辑层是系统核心，封装各模块规则：

- 用户与权限校验；
- 药材信息规范化处理；
- 方剂组成与配伍校验；
- 库存数量增减、阈值判断、预警生成；
- 操作日志记录；
- 统计分析与报表数据生成。

### 3.5 数据持久层

数据持久层通过 Django ORM 访问 MySQL，负责：

- 表结构映射；
- 业务数据存储；
- 索引优化；
- 数据备份与恢复。
  报告中已经明确要求创建中药材表、方剂表、药材-方剂关联表、库存流水表、用户角色表等核心数据表。

## 4. 功能架构设计

## 4.1 用户权限管理模块

该模块用于实现系统用户身份认证和角色授权。主要功能包括：

- 用户登录与退出；
- JWT 令牌签发与校验；
- 用户新增、删除、修改、禁用；
- 角色分配与权限绑定；
- 菜单权限与接口权限控制。

### 设计要点

- 登录成功后返回 Access Token 和 Refresh Token；
- 采用基于角色的访问控制（RBAC）；
- 前端根据角色动态加载菜单；
- 后端通过权限装饰器和中间件限制越权访问。

------

## 4.2 中药材信息管理模块

该模块用于管理中药材基础信息，是系统的基础数据中心。

### 主要功能

- 药材信息录入；
- 药材分类管理；
- 药材属性维护；
- 药材信息查询与筛选；
- 药材状态管理。

### 建议字段

- 药材编号
- 药材名称
- 别名
- 分类
- 性味
- 归经
- 功效
- 主治
- 产地
- 性状描述
- 储存方式
- 单位
- 参考价格
- 启用状态
- 创建时间、更新时间

### 设计亮点

结合开题报告中的文献综述，可将药材属性设计为“**固定字段 + 扩展属性 JSON**”的混合模型。这样既能满足常规检索，又便于后续扩展专业属性。

------

## 4.3 中药方剂管理模块

该模块用于维护方剂基本信息及其与药材的关联关系。

### 主要功能

- 方剂基础信息维护；
- 方剂组成药材配置；
- 药材剂量设置；
- 方剂适应症说明；
- 方剂查询与详情展示。

### 建议字段

**方剂主表**：

- 方剂编号
- 方剂名称
- 来源
- 功效
- 主治
- 用法用量
- 禁忌说明
- 备注

**方剂明细表**：

- 方剂ID
- 药材ID
- 剂量
- 单位
- 君臣佐使角色（可选扩展）
- 排序号

### 设计亮点

- 方剂与药材是多对多关系，应通过中间表维护；
- 后续可扩展“经典方”“院内方”“经验方”等标签；
- 可预留知识图谱或智能推荐接口，为论文后续“展望”部分埋下扩展点。
  这与开题报告中对方剂关联化管理的设计方向一致。

------

## 4.4 库存管理模块

库存管理是本系统最具业务价值的模块之一。

### 主要功能

- 药材入库；
- 药材出库；
- 当前库存查询；
- 安全库存阈值设置；
- 库存预警；
- 库存流水追踪；
- 盘点调整。

### 设计要点

- 每次入库、出库、盘点都必须写入流水表；
- 当前库存可以采用“库存主表 + 流水表”双表设计；
- 当库存数量低于安全阈值时自动生成预警记录；
- 预警信息可显示在首页仪表盘和库存预警页。
  报告中已明确要求实现库存出入库、预警与流水记录功能。

## 5. 数据库架构设计

## 5.1 核心实体关系

本系统数据库可围绕以下实体展开：

- 用户（User）
- 角色（Role）
- 权限（Permission）
- 药材（Herb）
- 方剂（Formula）
- 方剂明细（Formula_Item）
- 库存（Inventory）
- 库存流水（Inventory_Record）
- 预警（Inventory_Warning）
- 操作日志（Operation_Log）

### 5.2 ER 关系说明

1. 一个用户属于一个角色，一个角色对应多个用户；
2. 一个角色拥有多个权限，一个权限也可被多个角色拥有；
3. 一个方剂由多味药材组成，一味药材也可出现在多个方剂中；
4. 一味药材对应一条当前库存记录，但可对应多条库存流水；
5. 用户的关键操作会写入日志表。

### 5.3 推荐数据表

- `sys_user` 用户表
- `sys_role` 角色表
- `sys_permission` 权限表
- `sys_role_permission` 角色权限关联表
- `tcm_herb` 中药材表
- `tcm_formula` 方剂表
- `tcm_formula_item` 方剂药材明细表
- `inv_stock` 库存表
- `inv_record` 库存流水表
- `inv_warning` 库存预警表
- `sys_log` 操作日志表

### 5.4 索引建议

建议为以下字段建立索引：

- 药材名称
- 方剂名称
- 药材分类
- 库存状态
- 创建时间
- 用户名
- 角色编号

## 6. 接口架构设计

结合 RESTful 风格，接口可按资源进行划分。

## 6.1 认证接口

- `POST /api/auth/login` 用户登录
- `POST /api/auth/logout` 用户退出
- `POST /api/auth/refresh` 刷新令牌
- `GET /api/auth/profile` 获取当前用户信息

## 6.2 药材管理接口

- `GET /api/herbs` 药材分页查询
- `POST /api/herbs` 新增药材
- `GET /api/herbs/{id}` 查询药材详情
- `PUT /api/herbs/{id}` 更新药材
- `DELETE /api/herbs/{id}` 删除药材

## 6.3 方剂管理接口

- `GET /api/formulas` 方剂列表
- `POST /api/formulas` 新增方剂
- `GET /api/formulas/{id}` 方剂详情
- `PUT /api/formulas/{id}` 修改方剂
- `GET /api/formulas/{id}/items` 获取方剂组成

## 6.4 库存管理接口

- `GET /api/inventory` 当前库存查询
- `POST /api/inventory/inbound` 药材入库
- `POST /api/inventory/outbound` 药材出库
- `POST /api/inventory/check` 盘点调整
- `GET /api/inventory/records` 查询库存流水
- `GET /api/inventory/warnings` 查询库存预警

## 6.5 用户与权限接口

- `GET /api/users` 用户列表
- `POST /api/users` 新增用户
- `PUT /api/users/{id}/role` 分配角色
- `GET /api/roles` 角色列表
- `GET /api/permissions` 权限列表

## 7. 关键业务流程设计

## 7.1 登录认证流程

用户输入账号密码后，前端调用登录接口；后端校验成功后返回 JWT；前端保存令牌并跳转首页；后续请求在请求头中携带 Token；后端中间件解析 Token 并校验权限。

## 7.2 药材入库流程

药剂师选择药材并录入入库数量、批次、供应商、日期；系统校验字段合法性后，更新库存主表数量；同时写入库存流水表；若库存恢复到阈值以上则解除预警状态。

## 7.3 方剂维护流程

管理员或药剂师新增方剂主信息；选择组成药材并录入剂量；后端保存主表与明细表数据；方剂详情页可展示组成、功效、主治和用法。

## 7.4 库存预警流程

系统定时或在每次库存变更后比较当前库存与安全阈值；若低于阈值，则生成预警记录；前端在首页和库存预警页显示提醒。

## 8. 安全架构设计

根据开题报告，系统需要重点考虑传输安全、权限安全和数据安全。

### 8.1 身份认证安全

- JWT 认证；
- 密码加密存储；
- 登录失败次数限制；
- Token 过期与刷新机制。

### 8.2 接口访问安全

- 基于角色的接口权限校验；
- 防止越权访问；
- 参数白名单校验；
- 统一异常处理，避免敏感信息泄露。

### 8.3 数据安全

- MySQL 外键与约束控制；
- 后端 Serializer 双重校验；
- 定时备份核心数据；
- 操作日志可审计。

### 8.4 Web 安全

- HTTPS 传输；
- 防 SQL 注入；
- 防 XSS；
- 防 CSRF（若采用 Cookie 场景）；
- 文件上传类型校验。

## 9. 部署架构设计

结合开题报告中的部署要求，建议采用 Docker Compose 进行轻量部署。

### 部署组成

- `nginx`：反向代理与静态资源分发
- `frontend`：Vue 构建后的前端静态页面
- `backend`：Django API 服务
- `mysql`：业务数据库
- `redis`（可选）：缓存与验证码/会话支撑

### 部署优势

- 环境统一；
- 易于迁移；
- 便于本地开发与服务器上线；
- 便于后期扩展。

## 10. 本系统的可扩展设计

为了增强论文的完整性，建议在架构中预留以下扩展能力：

1. **中药材扩展属性**：使用 JSON 字段支持后续扩展药材专业属性；
2. **统计分析模块**：后续增加药材使用频次、库存消耗趋势分析；

5. **多端适配**：后续可扩展移动端小程序或 Pad 端。

## 11. 可直接写入论文的“架构设计总结”

本系统采用基于 B/S 模式的前后端分离架构，前端使用 Vue.js 实现用户界面与交互逻辑，后端基于 Django REST Framework 构建标准化 RESTful API，MySQL 作为核心关系型数据库实现业务数据持久化存储。系统围绕中药材管理、方剂管理、库存管理和用户权限管理四个核心模块展开，并辅以日志审计、安全控制和容器化部署机制，从而形成一个结构清晰、职责明确、扩展性良好的中医药管理系统技术架构。该架构既符合开题报告提出的研究目标，也能够满足后续系统实现、测试和论文撰写的需要。
